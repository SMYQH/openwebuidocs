### 企业级安全架构

对于企业组织来说，采用 AI 不只是技术决策，更是安全决策。每一次模型调用、每一份上传文档、每一个自动化工作流，都意味着有数据需要被保护。

Open WebUI 从设计之初就把安全放在核心优先级上。无论你面对的是严格的监管要求，还是希望保护组织的知识产权，Open WebUI 都提供了相应控制能力，帮助你更有信心地部署 AI。和所有自托管软件一样，最终负责环境、基础设施与配置安全的，仍是部署方自己。

关于聊天存储、管理员可见性、静态加密、临时对话、模型提供商暴露及推荐控制措施的详细说明，请参阅[聊天数据隐私与加密](/security/chat-data-privacy-and-encryption)。

### 安全架构

#### 你的数据，你的基础设施

与数据需要经过第三方服务器的 SaaS AI 平台不同，Open WebUI 让你对**整个数据管道拥有完全控制权**：

| 部署模式 | 说明 |
| :--- | :--- |
| **本地部署** | 完全部署在你自己的数据中心中。默认情况下，数据始终留在你的网络内部。 |
| **私有云** | 部署在你组织自己的云租户中（AWS、Azure、GCP），并完全掌控基础设施。 |
| **隔离网络 / Air-Gapped** | 为安全要求最严格的环境提供彻底的网络隔离。 |
| **混合部署** | 用灵活配置在可访问性与安全边界之间取得平衡。 |

对于处理敏感数据的组织——无论是专有研究、客户信息还是涉密材料——这样的架构都旨在帮助你维护整体安全态势。

如果你从主权 AI 的角度评估这一点，请首先定义你的部署必须控制的基础设施、数据、模型、身份、权限、日志和运维边界。请参阅[主权 AI 平台](/enterprise/sovereign-ai)清单，或 openwebui.com 上的[主权 AI 产品概述](https://openwebui.com/sovereign-ai)。

### 监管与合规考量

很多评估 AI 平台的组织，都受到 SOC 2、HIPAA、GDPR、FedRAMP 或 ISO 27001 等监管框架约束。Open WebUI 的自托管、本地部署与隔离网络架构，提供了可构成合规部署**一部分**的技术控制：

* **自托管数据管道**：除非你主动配置外部模型提供商，否则数据默认保留在你自己的基础设施内部。
* **身份集成**：支持 SSO、LDAP、RBAC，用于执行你的访问控制策略。
* **可审计日志**：容器原生日志流可与企业级 SIEM 工具兼容。
* **数据驻留控制**：由你决定数据的物理存储位置。

:::warning 合规责任归属
作为自托管软件，Open WebUI 部署并运行在你的基础设施中。合规认证适用于**你的部署环境**，而不是软件本身。由于部署、配置、访问控制和运行安全保障均由你管理，因此满足监管框架要求的责任也由你的组织承担。软件使用受 [Open WebUI License](/license) 约束。

通用最佳实践可参考我们的[加固文档](/getting-started/advanced-topics/hardening)。符合相应层级的企业客户还可获得部署与配置决策方面的实操支持。

### 身份与访问管理

与现有身份基础设施的无缝集成，可以在降低使用阻力的同时保持安全边界。

#### 企业身份集成

Open WebUI 可接入你组织已经在使用的身份系统：

* **LDAP 与 Active Directory**：直接连接你现有的目录服务，实现用户认证与管理。
* **单点登录（SSO）**：支持 SAML 与 OIDC 协议，让用户使用现有企业凭证访问 Open WebUI。
* **多因素认证（MFA）**：在主认证之上增加额外安全层。

#### 访问控制与权限

除了认证外，Open WebUI 还提供细粒度的用户权限控制：

* **基于角色的访问控制（RBAC）**：定义与组织结构对应的角色，以限制管理权限。
* **模型级权限**：控制哪些用户或用户组可以访问特定模型。
* **工作区隔离**：按团队或部门进行隔离，帮助限制未授权的数据访问。

### 数据治理

#### 审计与可追责性

对于受监管行业和重视安全的组织来说，系统使用可见性不是可选项。Open WebUI 支持标准企业日志模式：

* **基础设施级日志**：容器化架构允许将标准输出流直接接入你的日志基础设施（Splunk、Datadog、ELK）。
* **事件追踪**：跟踪 API 使用情况与系统事件，以监控异常。
* **保留策略控制**：由于数据库由你掌控，因此数据保留策略也由你决定，可按合规要求清除或归档。

#### 数据驻留

对于有地理位置数据要求的组织——无论是出于 GDPR、数据主权法律还是内部政策——Open WebUI 的自托管部署模式都能支持你将数据实际保留在政策要求的地点。


### 漏洞管理与披露

Open WebUI 中的安全问题通过一个单一的、明确定义的流程处理，而非临时处理：

* **单一权威渠道。** 漏洞通过 [GitHub Security Advisories](https://github.com/open-webui/open-webui/security) 进行报告和协调。报告由核心团队手动对代码进行分类和验证，而非自动分类。
* **修复、标识、披露。** 确认的问题会在实际可行的情况下尽快修复，分配 CVE 标识符（如适用），并在修复发布后以安全公告形式公开披露。完整历史记录可在 [Security Advisories 页面](https://github.com/open-webui/open-webui/security/advisories) 上审计：你可以查看发现了什么、如何解决的以及哪个版本包含了修复。
* **有争议的声明会得到公开回应。** 当外部 CVE 对软件或威胁模型进行错误描述时，团队会发布有理有据的[供应商立场声明](/security/vendor-dispositions)，而非让公开记录保持未质疑状态。

参与规则、受支持版本、威胁模型以及什么被视为或不被视为漏洞，均记录在[安全策略](/security/security-policy)中。

### 供应链与包完整性

Open WebUI 的已发布制品是从 [open-webui/open-webui](https://github.com/open-webui/open-webui) 仓库中的公开源代码构建的，并且可以从该源代码重现，因此任何关于发布的声明都可以对照代码进行验证。

自动化供应链扫描器（例如 socket.dev）会定期针对软件包提出行为风险指标、未确认的分类以及许可证或维护者评分。这些反映的是软件的设计能力及元数据产物，而非已确认的漏洞或恶意代码。如果你的审查中出现此类标记，[供应链与安全扫描器](/security/supply-chain-security) 解释了我们对每个类别的立场以及如何解读。第三方依赖项通过日常维护进行跟踪和更新。

### 代码执行与沙箱化

Open WebUI 中的服务器端代码执行受到明确的管理员操作控制，不会暴露给任意用户：

* **服务器端执行引擎（例如 Jupyter）是选择性加入的**，由管理员配置。默认情况下它们未连接到服务器运行时。
* **创作服务器端工具和函数被视为等同于授予服务器代码执行权限。** 它受 `workspace.tools` 权限限制，默认对非管理员禁用，并被记录为等同于向该用户授予对服务器的 shell 访问权限。
* **对于生产代码和 shell 执行，请使用 Terminals 编排器。** 与其暴露共享服务器运行时，[Terminals 编排器](/features/open-terminal/terminals) 在按会话、容器隔离的环境中运行执行，并具有可配置的[安全上下文和资源策略](/features/open-terminal/terminals/orchestration/policies)。这是企业部署的推荐模型；其他引擎的通用隔离指南请参见我们的[加固文档](/getting-started/advanced-topics/hardening)。

完整的模型请参见[工具和函数安全模型](/features/extensibility/plugin/tools)。

### 这对你的组织意味着什么

#### 对安全团队

Open WebUI 的代码库可公开审计，让你获得可见性与控制权；它会融入你现有的安全工具链，而不是制造新的盲区。如果在你的审查过程中，供应链或包扫描器（例如 socket.dev）标记了 Open WebUI 包，请参见[供应链与安全扫描器](/security/supply-chain-security)了解我们对这些指标的立场。

#### 对合规负责人

相关技术控制已具备，可支撑你的整体合规姿态。无论你是在准备审计，还是在回复安全问卷，Open WebUI 的架构都能提供支持你回答的技术依据。

#### 对 IT 管理层

用户管理不会演变成新的孤岛。Open WebUI 与你现有的身份基础设施协同工作，从而降低管理开销，并帮助你在各类工具之间保持一致的访问策略。
