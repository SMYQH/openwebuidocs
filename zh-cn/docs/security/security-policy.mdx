该政策的权威版本位于 [GitHub 安全政策](https://github.com/open-webui/open-webui/security)。本页面是其关键原则的快速参考。如有任何不一致，GitHub 版本优先。

我们的目标是保护 Open WebUI 的用户及其数据，并以清晰、一致且公开记录的方式处理安全报告。我们希望运营一个透明的安全流程，其中已接受的漏洞作为公告公开发布，以便任何人都可以看到发现了什么、如何解决的，以及最重要的是哪个版本包含补丁。

我们的立场：可见的公告历史记录是积极审查和有效披露流程的证据，而非衡量软件脆弱性的标准。

我们的技术架构和开发流程旨在最小化漏洞并维持利益相关者对我们的信任。我们采用自动化和手动技术的混合方法来跟上演变的威胁，重点是主动风险管理和负责任地处理任何报告的漏洞。

## 支持的版本

| 版本（分支） | 支持 |
| ---------------- | ------------------ |
| main             | :white_check_mark: |
| dev              | :x:                |
| 其他             | :x:                |

**如果某个问题在您提交时已在公开中修复或正在修复中，则该报告将不被接受**。它没有为发现或修复该问题做出贡献，我们不会为其发布公告。无论修复存在于哪个分支（包括 `dev`），也无论它是否在更早版本中已被静默解决，修复都被视为已存在。分支支持状态决定漏洞必须在何处**可重现**，而非修复是否已存在：在支持分支中存在的错误但已在 `dev` 中修复，在此规则下仍属于已修复问题。

这涵盖两种特定模式，我们均拒绝：

- 为**旧版本**中发现的、在当前支持版本发布时已解决的错误提交报告。
- **监控我们的公共提交或拉取请求**并为它们已处理或修复的问题提交报告。我们观察到对我们的公共提交和 PR 的自动化监控，产生了针对其他人已撰写的修复的报告；此规则的存在正是为了拒绝这种模式。

我们无需判断您是否独立发现了该问题。我们无法判断，而且这无关紧要。根据可证明的事实，您的报告重复了已公开且已修复或正在修复的工作；您严格来说是最后提交的，且无法区分独立发现与爬取。该问题的功劳属于首先发现或修复它的人，而此人如果公开披露而非先保密报告给我们，则也放弃了自己的权利。因此，公开披露的修复不会获得公告，也不会给任何人记功。

:::tip

**在报告之前，请确认您的发现是否仍在 `dev` 分支（及任何其他活跃开发分支）上可重现。** 我们公开开发，修复可能在发布前已提交到那里。先确认这一点可以节省您撰写报告的工作，而我们将不得不将其关闭为已修复。

:::

## 非漏洞的善意报告

如果您发现了您知道**根据本政策严格来说不是一个漏洞**，但公开披露仍然是不负责任的情况（例如，因为下游漏洞而需要紧急依赖升级），您**仍然可以通过 [GitHub Security Advisories](https://github.com/open-webui/open-webui/security/advisories/new) 私下报告**。我们会负责任地处理。

根据 CVE 规则，我们**不会**为此发布公告或生成 CVE，但我们**会**采取行动（例如发布升级）并保密处理该报告，直到问题解决。<ins>**如果因您的报告而发布了修复且您希望获得认可，我们将尽力致谢您（例如，作为更改的共同作者）。**</ins>

## 有效报告您能获得什么

如果您的报告描述了本政策下的真实漏洞，以下是您可以从我们这里获得的：

- **在公告中获得认可。** 您将被列为已发布公告的报告者。当多个报告者各自展示了不同的攻击向量时，每位都会获得认可（请参见[报告处理](#报告处理)）。
- **协调披露。** 在您与我们共同处理该问题时，我们不会在您不知情的情况下抢先发布。状态在公告本身上可见地更新，包括 CVE 请求，GitHub 会通知您这些更新，因此您可以全程跟进直至发布。
- **真正的修复，负责任地处理。** 对于具有广泛或严重实际影响的发现，我们可能会在修补版本发布后延迟最多约两周发布，以便管理员在细节公开前进行更新。

Open WebUI 由一个拥有明确安全流程所有权的小型核心团队领导。有效报告会得到认真审查、真正的修复、适当的认可，并通过 GitHub 公告记录进行协调处理。

## 与 CVE 计划的协调

**CVE 计划规则**（和 CNA 操作规则）是此处所有 CVE 处理的**基准**，本政策在其框架内运作。根据这些规则，报告是否构成 Open WebUI 中的安全漏洞由供应商决定；本政策记录了我们行使该决定的标准。在规则未明确说明的地方，它们仍然适用；在本政策指定我们如何将其应用于 Open WebUI 的地方，它是作为供应商发布的处置标准，而非对程序规则的替代或例外。

## 安全漏洞的报告位置和方式

:::info

Open WebUI 的社区之所以蓬勃发展，是因为像您这样的人，他们深切关心为每个人制造更安全的软件。

为确保您的发现确实有助于保护用户并迅速解决，请**仅**通过我们的[官方 GitHub 安全页面](https://github.com/open-webui/open-webui/security)提交所有安全漏洞报告。任何其他网站、服务或所谓的"赏金"平台都***不*与我们有关联**，您的重要工作根本无法到达能做出改变的人。

:::

**我们仅通过 [GitHub Security Advisories](https://github.com/open-webui/open-webui/security/advisories/new) 接受漏洞报告。** 通过**任何**其他平台提交的报告，包括但不限于第三方漏洞报告平台、漏洞经纪人、社交媒体、电子邮件、Discord 或 Reddit，将不予处理。

这不是程序性偏好。作为一个由小型核心团队领导的项目，Open WebUI 使用单一、权威的报告渠道进行安全工作，而 GitHub Security Advisories 正是该流程所在。我们不监控也不参与外部报告平台，通过这些平台到达的报告将被关闭且不予审查。

在其他平台上提交的报告在此处**没有效力**：它不赋予任何优先级，不建立提交日期，也不产生任何义务让我们进行分类、发布或以其他方式考虑。只有 GitHub Security Advisory 记录为本政策的目的而存在，包括确定谁先提交。

## 外国 CNA 和供应商处置

[基于多个外国 CNA 在未事先将报告通知我们即生成 CVE，和/或生成经不起审查的 CVE 的先例](./vendor-dispositions)，制定了此规则。

当报告通过 GitHub Security Advisories 提交且维护者根据本政策将其关闭为超出范围时，该关闭是该问题的**供应商处置**。CVE 编号机构（CNA）为此类问题生成 CVE 而未在结果记录中反映该供应商处置的行为，是违反供应商处置的行为。

我们通过以下方式回应此类记录：

1. 向 CVE 计划提交 **REJECT** 请求（以 **DISPUTED** 作为备选）
2. 在我们的[供应商处置](./vendor-dispositions)中公开编录该记录，注明发布 CNA
3. 拒绝提供供应商声明、版本映射、修复参考或任何其他会赋予该记录权威性的协调
4. 将来自单个 CNA 的重复模式上报至 CVE 计划 Root

**渠道合规不赋予 CNA 覆盖供应商处置的权利。** 在供应商处置已发布后，将已关闭的超出范围 GHSA 报告升级至第三方 CNA 的报告者同样被视为违反了供应商处置，并**可能被禁止未来的 GHSA 提交**。

## 报告指南

**[您可以在此处找到这些指南的权威版本。](https://github.com/open-webui/open-webui/security)**

我们感谢社区识别潜在漏洞的兴趣。如果您想报告不符合此处所述规则和指南的内容，您仍然可以报告，我们会处理（请参见上方的[非漏洞的善意报告](#非漏洞的善意报告)）。但是，我们**不会**接受低质量的漏洞报告。请确保提交的内容具有建设性、可操作性、可重现、文档完善，并遵守以下要求：

**安全边界：** 在本政策中，"安全边界"指我们承认的五个边界：机密性、完整性、可用性、真实性、不可否认性。我们广泛解释这些概念：其他安全框架中的等效概念也属于其中。有效漏洞必须跨越至少一个安全边界**且针对报告者以外的一方**。

1. **报告必须是漏洞**：安全漏洞是可利用的弱点，系统以非预期方式运行，允许攻击者绕过安全控制、获得未授权访问、执行任意代码或提升权限。配置选项、缺失功能和预期协议行为不是漏洞。漏洞必须跨越至少一个安全边界（定义如上）。

2. **无模糊报告**：诸如"我发现了漏洞"之类的提交，不包含任何细节，将被视为垃圾邮件且不被接受。

3. **需要深入了解**：报告必须反映对代码库以及 Open WebUI 使用方式的清晰理解，并提供关于漏洞的具体细节，包括受影响的组件和潜在影响。

4. **概念验证（PoC）是强制性的**：每个提交必须包括一个文档完善的概念验证，证明漏洞。PoC 必须展示（1）确切跨越了哪个安全边界，（2）漏洞如何被触发或滥用（输入、端点、UI 操作等），（3）攻击者现在可以执行哪些操作，以及（4）确切的重现步骤和命令（尽量可复制粘贴运行），附上预期结果与实际结果。如果保密是一个问题，我们鼓励报告者创建存储库的私人分叉并与维护者共享访问。缺少有效证据的报告可能被忽视。

5. **需要修复方案**：除 PoC 外，您必须提供**修复计划**（维护者可应用的可操作步骤）**或**补丁/PR。您的修复指导可以包括，例如：可能的根本原因（什么问题和在哪里）、要更改的位置（文件/模块/函数名称，如已知）、推荐的修复方法（验证/清理规则、身份验证检查、安全默认值等）以及任何安全权衡或要关注的潜在回归。

6. **默认配置测试**：漏洞报告必须使用 Open WebUI 的开箱即用默认配置进行测试和重现。仅在使用明确削弱的安全设置时出现的漏洞声明可能被丢弃。但是，如果您认为您已发现影响默认配置的安全问题，代表对预期安全控制的真正绕过，或仅适用于非默认配置但所讨论的配置可能被生产部署使用，那么我们绝对想听到它。此政策旨在过滤配置问题和部署问题，而非劝阻合法的安全研究。

7. **需要威胁模型理解**：报告必须展示对 Open WebUI 的自托管、单租户、经过身份验证、可扩展、基于角色的访问控制架构的理解。在不承认架构差异的情况下将 Open WebUI 与具有根本不同安全模型的服务进行比较可能导致报告被拒绝。

8. **CVSS 评分准确性**：您不必包含 CVSS 分数。如果您将 CVSS 部分留空，我们会在发布前为您填写。如果您确实包含 CVSS 分数，它必须根据 CVSS 方法准确反映漏洞；我们会调整不准确的分数。如果您引用其他 CVE 来支持您的报告，确保它们在漏洞类型、威胁模型和攻击向量上**确实可比较**。

9. **管理员操作不在范围内**：需要管理员主动执行不安全操作的漏洞不被视为有效漏洞。管理员拥有完整的系统控制权，应理解其操作和配置的安全含义。这包括但不限于添加恶意外部服务器（模型、工具、webhook、函数）、将不受信任的代码粘贴到函数/工具中，或故意削弱安全设置。需要管理员疏忽或对管理员进行社会工程的报告可能被拒绝。但是，如果您认为您已发现影响管理员的漏洞且并非由管理员疏忽或故意恶意操作引起，那么我们绝对想听到它。此政策旨在过滤对管理员的社会工程攻击和类似行为，而非劝阻合法的安全研究。

10. **工具和函数代码执行是预期行为**：Open WebUI 的工具和函数功能**被设计为**在服务器上执行用户提供的 Python 代码。这是核心的、有意的功能，而非漏洞。函数创建仅限于管理员。工具创建由 `workspace.tools` 权限控制，该权限**为非管理员用户默认禁用**，只应授予完全信任的用户——他们在信任方面等同于系统管理员。**给予用户创建工具的能力相当于给他们对服务器的 shell 访问权限。** 更一般地说，**描述涉及工具或函数的任何攻击链——包括但不限于代码执行、文件访问、网络请求或环境变量访问——的报告将作为非漏洞/预期行为关闭。** 这适用于直接代码执行和 frontmatter 包安装（`pip install`）。

11. **遗留代码路径不在范围内**：Open WebUI 维护一些在官方文档中明确标记为**遗留**的代码路径。遗留路径保持可用——有时仍然是默认值——纯粹出于向后兼容性原因，而非因为它们是受支持或维护的表面。安全和功能工作发生在受支持的替代品上，而非遗留路径。描述**仅在遗留代码路径上且不在受支持替代品上重现的**安全边界问题的报告在该规则下通常超出范围。如果问题在遗留路径**和**受支持的现代替代品上都重现，或遗留路径是实现给定功能的唯一记录方式（尚无迁移目标），我们仍然想听到它。

12. **AI 报告透明度**：由于漏洞报告数量的激增，您必须披露是否以任何方式使用了 AI——无论是编写报告、生成 PoC 还是识别漏洞。AI 辅助的漏洞报告**默认不会被拒绝**，但未声明 AI 使用但看起来是 AI 辅助的报告将受到更严格的审查。

13. **自我影响问题不是漏洞**：漏洞需要跨越影响**报告者以外的一方**的安全边界。仅针对报告者自己的数据、账户、会话或环境跨越五个公认的安全边界之一不是漏洞——它是一个错误，属于[问题跟踪器](https://github.com/open-webui/open-webui/issues)，而非安全报告。如果相同的操作也影响另一用户、操作者、主机系统或共享资源，请在 PoC 中清楚地标识该第二方。

不符合要求的提交可能被关闭，重复或极端违规者可能被禁止。我们的目标是培养一个建设性的报告环境，其中高质量的提交促进所有用户的安全性提升。

## 预期时间框架

我们旨在及时分类新报告、发布修复和公告。报告按有效性、严重性、可重现性和用户影响进行优先级排序，与发布和核心项目维护并行。Open WebUI 由一个小型核心团队领导，安全报告通过该核心流程处理。

**请预期数周**时间用于您的报告被分类、调查、修复和发布。当有实质性进展可报告时，公告上会发布状态更新。公告在整个分类、调查、修复和发布过程中保持为权威信息来源。

**我们不接受报告者强加的发布时间表。** Open WebUI 根据验证、修复、发布协调和用户影响来设置披露时间。外部强加的硬性截止日期不会改变报告优先级、公告时间或发布协调。附在报告上的截止日期不会改变其处理的时间或速度。

对于我们认为具有**广泛或严重实际影响**的发现——无论 CVSS 分数如何——我们可能会在修补版本发布后延迟**几天到最多约两周**发布，以便管理员有时间更新其实例。

## 报告处理

当多个独立报告者描述同一漏洞类别但每个都展示了**不同且独立的利用向量**时——例如，通过不同端点到达的相同缺失授权检查——我们会将它们合并到最早的提交中，**并在合并后的公告中对每个展示了不同路径的报告者给予认可**。仅会为合并的公告发布一个 CVE。

另一种情况：如果您报告了一个他人之前已报告过的有效漏洞（相同的漏洞，相同的利用向量），我们将关闭您的报告作为重复。最早的提交是我们将继续处理的，我们不会为同一漏洞发布多个公告。

### 为什么重复报告不获得认可

我们仅认可给定漏洞的最早提交者：

1. **第一份报告完成了工作。** 当后续报告到达时，分类和修复已在进行中。后续报告不改变结果或时间线；认可它们会歪曲是什么推动了修复。
2. **为重复提供奖励会鼓励提交泛滥。** 如果相似但较晚的提交也能获得认可，理性的做法是浏览公开公告并提出变体。我们已经看到这种压力：先提交规则正是限制它的原因。
3. **共同发现不同于重复。** 多个报告者**在一份公告上获得认可**当每个贡献了**独特的**发现时——不同的向量、不同的受影响组件、早期提交未覆盖的不同子路径。这是上面的合并规则。提交现有报告的重复不是共同发现。

## 保密披露

通过 GitHub Security Advisories 提交的漏洞报告是**私密和保密的**。在公告**完全发布**之前——不仅仅是分配了 CVE ID 时，而是公告本身公开可见时——**公开披露**与已提交漏洞报告相关的**任何**细节是**严格禁止的**。

此禁止适用于**所有渠道**，包括但不限于拉取请求、问题或讨论上的评论（在 GitHub 或其他地方）、社交媒体（Discord、Reddit 或任何其他平台）、博客、论坛或任何其他网站或服务。

过早披露破坏了所有 Open WebUI 用户的安全**并违反了**负责任披露流程中固有的信任。**在官方发布前过早公开披露漏洞细节的报告者<ins>将被永久禁止进行未来的报告。</ins>**

## 非漏洞相关的安全问题

如果您的关注点不是漏洞但仍与安全相关，请改用以下渠道：

## 工具、函数和管道安全

Open WebUI 通过**工具**、**函数**（包括管道、过滤器和操作）和**管道**提供强大的可扩展性。这些功能允许您使用自定义 Python 代码扩展 Open WebUI 的功能。但是，这种力量伴随着安全责任。以下每个文档页面都包含其自己的安全警告 — 查看它们以获得完整的图片：

- [插件安全概览](/features/extensibility/plugin/) — 关键安全警告、信任模型和工作区访问限制
- [工具](/features/extensibility/plugin/tools/) — 工具创建、导入和 `workspace.tools` 权限的安全警告
- [函数](/features/extensibility/plugin/functions/) — 仅管理员创建、代码执行警告
- [管道](/features/extensibility/pipelines/) — 管道插件系统的任意代码执行警告

:::warning

**工具、函数和管道在您的服务器上执行任意 Python 代码。** 这是有意的 — 这是使它们强大的原因。但是，这意味着它们具有与 Open WebUI 后端流程相同的访问级别。

:::

### 安全影响

当您安装工具、函数或管道时，您授予它执行以下操作的能力：

- **访问文件系统** — 读取或写入后端流程可以访问的任何文件
- **进行网络请求** — 连接到外部服务，可能泄露数据
- **执行系统命令** — 通过子流程运行 shell 命令
- **访问环境变量** — 读取 API 密钥、秘密和配置
- **修改数据库** — 访问或改变存储的数据
- **消耗计算资源** — 运行 CPU 密集型操作

### 最佳实践

| 实践 | 描述 |
|----------|-------------|
| **仅从受信任的源安装** | 仅使用来自官方社区库或您信任的源的工具/函数 |
| **安装前检查代码** | 在导入工具/函数之前阅读并理解其功能 |
| **限制工作区访问** | 仅向受信任的管理员授予工作区权限 |
| **审计已安装的插件** | 定期查看已安装的工具（工作区 → 工具）和函数（管理面板 → 函数） |
| **保护您的数据目录** | `/app/backend/data` 目录包含您的数据库和缓存的插件——保护它免受未授权访问 |
| **监控资源使用** | 观察可能表示加密货币挖掘或其他滥用的意外 CPU 尖峰 |
| **使用官方 Docker 镜像** | 仅从 `ghcr.io/open-webui/open-webui` 或 `openwebui/open-webui`（Docker Hub）拉取——非官方镜像可能被破坏 |

### 什么不构成漏洞

以下场景**不**被视为漏洞，因为它们需要管理员操作（另请参见上方报告指南中的[管理员操作不在范围内](#报告指南)和[威胁模型理解](#报告指南)）：

- 管理员安装了恶意的工具或函数
- 管理员向不受信任的用户授予工作区访问权限
- 管理员从不受信任的来源导入代码
- 攻击者拥有数据卷的写权限并注入恶意插件

更一般地说，**任何涉及工具或函数的攻击链报告，包括但不限于代码执行、文件访问、网络请求或环境变量访问，都将以非漏洞/预期行为为由关闭。** 如果管理员向不受信任的用户授予工具权限，这构成了有意的错误配置。

这些场景代表的是**管理员疏忽**或**环境受损**，而非 Open WebUI 本身的漏洞。详情请参阅[我们的安全政策](https://github.com/open-webui/open-webui/security)和[插件安全文档](/features/extensibility/plugin/)。

## 生产部署安全

Open WebUI 为私有、受信任的网络设计，默认配置针对易于设置进行了优化。处理敏感数据的生产部署应查阅[加固指南](../getting-started/advanced-topics/hardening.md)以获取完整清单。

一个关键注意事项：**在没有 Redis 的情况下，登出和修改密码操作不会撤销现有的 JWT 令牌。** 它们会保持有效直到过期（默认：4 周）。这意味着泄露的令牌无法被废弃，且管理员发起的账户停用不会立即阻止访问。对于生产部署，建议[配置 Redis](../tutorials/integrations/redis.md) 或缩短 `JWT_EXPIRES_IN` 以限制暴露窗口。详情请参阅[令牌撤销](../getting-started/advanced-topics/hardening.md#token-revocation)。

- 对我们的架构和流水线进行内部及定期的外部审查
- 前端和后端的自动化及手动测试
- 作为持续开发的一部分，进行主动风险管理和代码审查
- 持续改进并集成静态/动态分析的高级工具
