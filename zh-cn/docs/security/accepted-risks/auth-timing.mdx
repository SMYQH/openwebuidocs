# 认证时序侧信道

|  |  |
| :--- | :--- |
| **类别** | CWE-208，可观察时序差异 |
| **状态** | 可利用信号已缓解；亚毫秒级残余已接受 |
| **追踪为** | [GHSA-7rw5-9f7q-xj36](https://github.com/open-webui/open-webui/security/advisories/GHSA-7rw5-9f7q-xj36) |

---

## 概述

登录端点对已有账户的响应时间明显长于不存在的账户，因为密码哈希仅在账户存在时被验证。差异（约 150 毫秒对比约 5 毫秒）使攻击者可以通过响应时间枚举有效账户，从而为后续密码喷洒攻击提供基础。

## 我们缓解的措施

此问题已报告给我们，并追踪为 [GHSA-7rw5-9f7q-xj36](https://github.com/open-webui/open-webui/security/advisories/GHSA-7rw5-9f7q-xj36)。我们在登录路径中解决了该问题：认证现在在未找到用户或未找到活跃凭证的路径上，对一个固定的虚拟哈希执行 bcrypt 比较。因此，无论账户是否存在，每次登录尝试都支付相同的 bcrypt 开销。

经测量，这将差异从约 20-40 倍缩小到约 1.03 倍（bcrypt 对比 bcrypt），这与常规的运行间方差无法区分。实际可利用的枚举预言机已被移除。

剩余的毫秒级残余按原样接受。这不是进一步努力的问题：剩余的部分受限于物理和运行时限制（数据库、Python 解释器、CPU 行为、操作系统调度和网络抖动），而非 Open WebUI 代码可以改进的内容，因此从这里再难有更多收获。推理在下方[剩余部分及为何不可修复](#剩余部分及为何不可修复)中阐述。

## 剩余部分及为何不可修复

完美的恒定时间认证在应用级代码中无法实现。一旦主要的 bcrypt 开销被均等化，剩余差异为亚毫秒级，由 Open WebUI 无法控制的层次产生：

- **数据库查找**（已有电子邮件的索引命中对比不存在电子邮件的未命中），
- **Python 解释器和垃圾收集器**，
- **CPU 行为**：缓存状态、分支预测、推测执行，
- **操作系统调度**，
- **网络抖动**。

这些都无法从 Python 中驱动到完全为零。固定时间填充（"总是在设定的窗口内响应"）也无法消除它：窗口必须填充到最坏情况，这会惩罚每个合法登录，且足够精确的攻击者仍然可以观察到填充开始之前完成的工作。它用真实的永久成本换取了虚假的完整性感。

残余信号为亚毫秒级，且受网络和调度噪声主导，因此无法在网络中可靠测量。我们因此将其接受为已知限制，而非可修补的缺陷。
