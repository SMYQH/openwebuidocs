# 已知与已接受风险

**Open WebUI 已评估、尽可能缓解并在技术上正式接受的残余风险。**

某些弱点无法在应用代码中完全消除。它们受限于编程语言、运行时、编译器和解释器优化、操作系统、CPU 或网络，而非 Open WebUI 可以修补的缺陷。对于这些，我们将可利用部分缩小到应用级代码允许的范围，然后将剩余部分记录在此，作为已知的已接受风险。

每条记录说明了风险是什么、我们做了什么来缩小它、剩余部分为何无法由我们修复，以及它如何符合我们的[威胁模型](#威胁模型)。

---

## 为何接受这些风险

接受残余风险是有意的工程决策，而非疏忽。仅当以下两个条件都成立时，弱点才会被列入此处：

- **可利用部分已尽可能缓解**在应用级代码允许的范围内，剩余的**残余部分是有边界的**，在实践中不可靠利用。
- **消除该残余部分是不值得的**，原因至少符合以下之一：
  - 它**在技术上无法由我们修复**，源于语言、运行时、操作系统、CPU 或网络，而非 Open WebUI 自身逻辑；或
  - 进一步修复会带来**真实且不成比例的成本**，降低或移除用户合法依赖的功能，或损害性能或正确性，以换取边际或虚幻的安全收益。

无论哪种情况，我们都会明确记录残余及其限制，而不是暗示无法或不应做出的保证。

---

## 威胁模型

Open WebUI 是一个**自托管、单租户、经过身份验证、基于角色**的应用程序。此处的风险在此模型内评估：

- 所有用户在访问受影响功能前均经过**身份验证**。
- **管理员**是受信任的行为者，拥有完整的系统控制权。
- 报告根据**默认或更严格**的配置进行评估。

如果残余的实际影响主要由网络和调度噪声主导，或仅在威胁模型已排除的条件下可观察，则该残余被视为已接受而非可修补的缺陷。有关完整模型，请参阅[安全策略](../security-policy)。
