# 供应链与安全扫描器

自动化软件组成和供应链扫描器（例如 socket.dev）会定期使用风险指标标记 Open WebUI 的 PyPI 包和容器镜像。本页面是 Open WebUI 团队对这些指标的官方立场，以便评估 Open WebUI 的安全团队能够准确解读它们。

:::info 简短版
这些扫描器对 Open WebUI 提出的指标均不对应于已命名的漏洞、CVE 或已确认的恶意代码。大多数是自动化的行为启发式方法，标记软件按设计具有的能力。有些更进一步，发布令人担忧且未经证实的声明，最明显的是"潜在恶意软件"横幅，而未识别任何特定文件、函数或代码行，也未提供任何证据支持该声明。Open WebUI 中未发现任何恶意代码。由于每个发布的工件都是从 [open-webui/open-webui](https://github.com/open-webui/open-webui) 中的公开源代码构建并可重现，因此每项声明都可以直接对照代码进行验证。
:::

## 这些扫描器衡量什么

供应链扫描器根据自动化信号对包进行评分：它使用哪些语言特性、它可以访问哪些系统功能、其依赖项是什么样的以及基于模式的分类器。这些信号描述了一个包**能够**做什么，而不是它是否**滥用**了任何东西。对于一个自托管的 AI 平台，得分最高的能力（运行代码、访问网络、调用 shell）恰恰是产品的核心、有文档记录的功能。因此，高信号计数反映了软件的功能，而不是其中的缺陷。

扫描器显示的标题数值分数 — 例如，满分 100 的供应链分数 — 是这些相同自动化信号的聚合，因此低于 100 的分数反映了下面描述的特征标志，而不是已确认缺陷的计数。

我们审查每个新标志，但标志是分析的起点，而不是发现。

## 各类指标

### 行为能力标志："使用 eval"、"shell 访问"、"网络访问"

这些描述了预期的、有文档记录的功能：

- **网络访问**是 Open WebUI 与模型提供商和外部工具通信的方式。
- **代码执行（"eval"）和 shell 访问**是代码解释器以及工具和函数系统，它们按设计执行代码。

这些能力由管理员配置和基于角色的权限控制（请参见下面的[代码执行和沙箱](#代码执行和沙箱)）。扫描器标记的是能力，而非任何滥用，因此这些都不是漏洞。

### 未经证实的分类："潜在恶意软件"、"AI 检测的安全风险"、"潜在漏洞"

这些是最可能引起审查人员警觉的指标，也是最缺乏证据支持的：

- **"潜在恶意软件"**是在未识别任何特定文件、函数或代码行且未附上任何支持证据的情况下断言的。Open WebUI 中未发现任何恶意代码。在我们看来，发布"检测到恶意软件"横幅而不指明代码或展示证据，在实质上是误导性的，特别是对于企业审查人员——他们合理地认为这样的横幅是已确认的发现，而非自动化猜测。
- **"AI 检测的安全风险"**是由上述行为信号驱动的机器生成分类（该软件"可能包含潜在安全问题"）。它未指明任何特定代码，也未确认任何内容。
- **"潜在漏洞"**，根据扫描器自身的措辞，是一个*待进一步分析和确认*的项目。那是一个开放的队列条目，而非发现。

由于 Open WebUI 发布的包完全从公开源代码构建，任何这些声明都可以对照代码进行验证，我们会在出现未经证实的横幅时请求与扫描供应商重新审查。

### 依赖项警报和"混淆代码"

这些源于传递性依赖项以及分发包中捆绑的预构建、压缩前端资源。压缩的 JavaScript 通常被扫描器分类为"混淆"，因为它不可人工阅读，但它是普通的编译前端代码，而非 Open WebUI 源代码中手工混淆的逻辑。依赖项警报追踪更广泛的生态系统，并通过常规依赖项维护进行处理。

容器镜像 CVE 是另一回事：那些源于基础镜像，通过基于更新基础重建来解决，不属于此处讨论的 PyPI 包扫描的一部分。

### 维护者数量

扫描器报告从包元数据中获取的维护者数量，对 Open WebUI 通常为"1 个维护者"。该数字是 PyPI 包上具有发布权限的账户数量——一个单一的发布账户——而非开发项目的人数。Open WebUI 由一个核心团队和广泛的外部贡献者基础构建，这在存储库的[贡献者历史](https://github.com/open-webui/open-webui/graphs/contributors)和我们的[团队页面](/team)上是可见的。从一个账户发布版本是常见且有意的打包实践，而非维护者 Bus Factor 为 1。

## 代码执行和沙箱

评估者经常询问 Open WebUI 是否"默认在没有强制沙箱的情况下"执行代码。关键在于：服务器端代码执行被明确的管理员操作所门禁，而非向任意用户开放：

- **服务器端执行引擎（如 Jupyter）是可选的**且由管理员配置。它们默认未连接到服务器运行时。
- **创作服务器端工具和函数被视为等同于授予服务器代码执行权限。** 它受 `workspace.tools` 权限限制，该权限**为非管理员用户默认禁用**，授予它在文档中被记录为等同于向该用户授予服务器的 shell 访问权限。
- **隔离执行环境是有文档的部署责任。** 在管理员启用服务器端执行引擎的情况下，在适当的容器、网络和文件系统边界内运行它，涵盖在我们的[加固指南](/getting-started/advanced-topics/hardening)中。

这是有意的分层设计：服务器端代码执行被明确的管理员配置和默认关闭的权限所门禁，而非对每个用户可用。有关完整模型，请参见[工具和函数安全模型](/features/extensibility/plugin/tools)和[安全策略](./security-policy)。

## 给企业评估者的建议

Open WebUI 由一个核心团队积极维护，该团队运行定义明确、动手操作的安全流程，而非依赖自动化评分：

- **每份报告都由人工审查。** 传入的安全报告由人工根据实际代码进行分类和验证。没有任何内容是自动关闭或自动分类的。
- **有效发现得到妥善修复和披露。** 我们在合理可能的情况下尽快修补已确认的问题，为有效发现请求 CVE 标识符，并在修复发布后以 GitHub 安全公告的形式公开结果。
- **记录是公开和可审计的。** 我们已接受、修复和发布的问题的完整历史记录可在 [Open WebUI 安全公告页面](https://github.com/open-webui/open-webui/security/advisories)上查看。任何人都可以看到发现了什么、如何解决的以及哪个版本包含修复。
- **无效或特征错误的声明会得到回应，而非被忽略。** 当外部 CVE 错误描述软件或我们的威胁模型时，我们会发布有充分理由的[供应商处置](./vendor-dispositions)，而非让公开记录不受质疑。
- **一个权威渠道。** 所有漏洞处理都通过 GitHub Security Advisories 进行，由团队拥有验证、修复和披露时间的控制权。

综合来看，这比自动化风险评分是一个实质上更强的信号：一个公开、可审计的已发现、已修复和已发布问题的追踪记录，由人工端到端运行的流程产生。我们乐于直接支持您的评估。

有用的起点：

- [安全策略](./security-policy)：报告流程、威胁模型以及什么被视为什么不被视为漏洞。
- [供应商处置](./vendor-dispositions)：我们对外部报告的特定 CVE 的正式评估。
- [工具和函数](/features/extensibility/plugin/tools)：代码执行和权限模型。

如果您认为扫描器标志对应的是真实安全问题而非行为启发式方法，请通过 [GitHub Security Advisories](https://github.com/open-webui/open-webui/security) 报告，以便进行验证并在确认后发布。
